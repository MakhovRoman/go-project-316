## hexlet-go-crawler

Веб-краулер для анализа структуры сайта и SEO-параметров.

## Использование

```
hexlet-go-crawler [флаги] <url>
```

## Флаги

| Флаг | По умолчанию | Описание |
|------|-------------|----------|
| `--depth` | `10` | Глубина обхода (см. ниже) |
| `--timeout` | `15s` | Таймаут одного запроса (например, `5s`, `500ms`) |
| `--retries` | `1` | Количество повторных попыток при ошибке |
| `--delay` | `0` | Задержка между запросами (например, `200ms`, `1s`) |
| `--rps` | `0` | Ограничение запросов в секунду (переопределяет `--delay`) |
| `--user-agent` | — | Произвольный заголовок User-Agent |
| `--workers` | `4` | Количество параллельных воркеров |

## Глубина обхода

Флаг `--depth` определяет максимальное количество переходов по ссылкам от стартового URL.

- Стартовая страница всегда имеет `depth = 0` в отчёте.
- Её прямые ссылки — `depth = 1`, ссылки на следующем уровне — `depth = 2`, и так далее.
- Когда глубина страницы достигает значения `--depth`, её исходящие ссылки **не добавляются** в очередь обхода.

Примеры:

| `--depth` | Что обходится |
|-----------|---------------|
| `1` | Только стартовая страница |
| `2` | Стартовая страница и её прямые ссылки |
| `3` | Стартовая страница и два уровня ссылок |

Обходятся только страницы в пределах домена стартового URL. Внешние ссылки проверяются на доступность и фиксируются как битые при статусе отличном от 200, но не обходятся.

## Скорость обхода

Краулер поддерживает два способа ограничить частоту HTTP-запросов.

**Фиксированная задержка** — флаг `--delay`:

```
hexlet-go-crawler --delay=200ms https://example.com
hexlet-go-crawler --delay=1s https://example.com
```

**Целевой RPS** — флаг `--rps` (запросов в секунду):

```
hexlet-go-crawler --rps=5 https://example.com
```

Если указаны оба флага, `--rps` имеет приоритет. Задержка применяется глобально — между последовательными обходами страниц, а не внутри отдельных потоков. При отмене контекста ожидание прерывается немедленно.

Без ограничений краулер работает на максимальной скорости.

## Повторные попытки

Флаг `--retries` задаёт максимальное количество повторных попыток при временных ошибках.

```
hexlet-go-crawler --retries=3 https://example.com
```

Повтор выполняется при следующих условиях:
- сетевая ошибка (нет соединения, таймаут)
- HTTP 429 Too Many Requests
- HTTP 5xx (500, 502, 503, 504 и другие серверные ошибки)

Постоянные ошибки (404, 403, 401 и другие 4xx) повторно не запрашиваются.

Между попытками краулер делает паузу: если задан `--delay` или `--rps` — используется их значение, иначе фиксированная задержка 100ms. При отмене контекста ожидание между попытками прерывается немедленно.

После исчерпания лимита результат последней попытки фиксируется в отчёте как ошибка.

## Ассеты

Для каждой страницы в отчёт добавляется поле `assets` — массив ресурсов, на которые ссылается HTML.

Какие теги парсятся:

- `<img src="...">` → `type: "image"`
- `<script src="...">` → `type: "script"`
- `<link rel="stylesheet" href="...">` → `type: "style"`

Структура каждого элемента:

```json
{
  "url": "https://example.com/static/logo.png",
  "type": "image",
  "status_code": 200,
  "size_bytes": 12345,
  "error": ""
}
```

Все пять полей присутствуют всегда — даже при ошибках. При сетевой ошибке `status_code = 0`, при HTTP `>= 400` — соответствующий код плюс описание в `error`.

**Размер ассета** определяется так:

1. Если сервер вернул заголовок `Content-Length` — используется его значение.
2. Иначе размер считается по фактически прочитанному телу ответа (с лимитом 10 МБ).
3. Если тело прочитать не удалось — `size_bytes = 0`, причина пишется в `error`.

**Кэш ассетов**: если один и тот же URL встречается на нескольких страницах, сетевой запрос выполняется один раз — все последующие обращения возвращают кэшированный результат, поэтому данные в отчёте совпадают.

Ассеты используют тот же контракт `--retries` / `--delay` / `--rps`, что и страницы.
